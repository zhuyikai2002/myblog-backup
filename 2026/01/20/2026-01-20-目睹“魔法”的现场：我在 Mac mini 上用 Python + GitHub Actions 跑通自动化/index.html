目睹“魔法”的现场：我在 Mac mini 上用 Python + GitHub Actions 跑通自动化

发表于2026-01-20|更新于2026-01-28|技术日记
|浏览量:

摘要：我坐在旁边亲眼看着老哥把自动化玩出了“魔法”——写脚本、改 Bug、接机器人推送，一气呵成。被刺激到的我，索性换阵打开吃灰的 Mac mini，硬着头皮从 C 切到 Python 自动化。一路踩了 403、Git 权限（Exit code 128）、数据为空导致脚本崩盘的坑，但这次我没让它烂尾：cloudscraper 过墙、YAML 加防崩、PAT 解决权限，最后 Actions 全绿、手机收到推送。更爽的是收尾：Cursor 直接帮我把 README 一次性写成中/英/日/阿四个版本，还自动操控浏览器把 Release 1.0 发了出去。

时间： 2026.01.20
关键词： Python / GitHub Actions / 自动化 / Mac mini / Cursor
状态： 一次没烂尾的折腾

🧪 1) 缘起：目睹“魔法”的现场

今天发生了一件对我触动很大的事。

我就坐在旁边，眼睁睁看着我那个老程序员大哥，给他的 GitHub 仓库配一个“青龙面板 + 创想三维 3D 打印机自动签到”的脚本。

他那一套操作简直太丝滑了：写了个 Shell 脚本，发现需要 Python，转手丢给 ChatGPT 转码；转出来有 Bug，他眉头都没皱，三下五除二就修好；最后直接通过钉钉机器人发来了“签到成功”的推送。

那一瞬间，我手里的富士笔记本（Linux Mint 主力机）突然就不香了。

我满脑子都是：我也要整一个！我也要这种“掌控感”！

🍎 2) 换阵：重启那台“画图娱乐机”

为了这份仪式感，我做了一个违背祖训（bushi）的决定：放下我平时写 Blog、学技术的富士 Linux 笔记本，打开了那台平时只用来娱乐的 Mac mini。

说起这台 Mac mini，它的身世挺“坎坷”的：

  • 上次高强度工作，还是我用它跑 Fusion 360 设计 DIY 蓝牙音箱外壳（虽然后来项目搁置了）
  • 上面的 PyCharm 和 CLion，还是两三个月前我刚接触技术时，为了部署一个类似“球球大作战”的游戏 VPS 下载的（最后也因为各种报错烂尾了）

今天看着图标上落满的“电子灰尘”，我决定：就在这台曾经“失败”过的电脑上，重新开始。

🧱 3) 实战：从 C 到 Python 的跨越

平时我都是缠着 AI 写 C 语言，折腾 STM32 这种嵌入式硬件。

但今天受了老哥的刺激，我决定挑战 Python 自动化。

这一路依旧是“步步惊心”：

  • 第一关：爬虫被拦：刚写好的脚本，直接被服务器防火墙拦截，喜提 403 Forbidden。
  • 第二关：Git 噩梦：以前在 Linux 上用 Git 顺风顺水，今天在 Mac 上配 GitHub Actions，死活报 Exit code 128。
  • 第三关：逻辑崩盘：因为没抓到数据，文件生成失败，脚本直接报错退出。

🔧 4) 破局：这一次，没有烂尾

两三个月前，我在 Mac 上折腾游戏服务器，因为报错放弃了；设计蓝牙音箱，因为各种原因搁置了。

但今天，在 AI 的辅助下，我死磕到底：

  • 引入 cloudscraper，伪装成浏览器，绕过了防火墙
  • 修改 GitHub Actions 的 YAML，加上“数据为空/文件不存在”的防崩判断
  • 用 Personal Access Token（PAT）解决 Git 权限问题

当 GitHub Actions 的界面终于从红色变成全绿，手机微信收到“博客更新”推送的那一刻，我感觉我不仅是在模仿老哥，更是在战胜这台电脑上曾经那个“半途而废”的自己。

⚡ 5) 加速器：Cursor 一口气给我补齐“文档 + 发版”

折腾到这里，按理说就可以收工了。

但我突然意识到：只要项目开始有第二个人看（哪怕只是未来的我），就一定得把“怎么用”说清楚。否则代码跑得再通，也只是我自己的私房菜。

于是我干了两件以前我经常拖着不做、最后导致烂尾的事：写 README、发 Release。

5.1 四国语言 README（中 / 英 / 日 / 阿）

我把需求丢给 Cursor：

  • 要有中文主 README
  • 再给我 English、日本語、العربية 四个版本
  • 内容要一致：项目简介、环境要求、快速开始、Actions 说明、常见问题

它给我的不是“翻译腔”的直译，而是像真的开源项目一样把结构搭好了：每种语言都有清晰的标题层级和使用步骤。

那一刻我第一次感觉，写文档也能像写代码一样“有脚手架”。

5.2 自动操控浏览器：发布 Release 1.0

更离谱的是，我本来还准备手动点 GitHub 网页，结果 Cursor 直接自动操控浏览器窗口：

  • 打开仓库的 Releases 页面
  • 新建 Release
  • 填写版本号（1.0）、标题与说明
  • 点击发布

当 Release 1.0 出现在页面上时，我突然理解了那种“掌控感”的来源：不是因为我一行不差地手搓了所有东西，而是我知道每一步应该发生什么，并且把它真的推到终点。

🌙 6) 尾声

看着 Mac 屏幕上绿色的 Success 图标，我突然觉得：这台电脑不再只是画图和娱乐的工具了。

虽然那个“球球大作战”游戏没成，虽然蓝牙音箱还没做完，但今天我用 Python 和 GitHub Actions 证明了——只要死磕，代码总是能跑通的。
