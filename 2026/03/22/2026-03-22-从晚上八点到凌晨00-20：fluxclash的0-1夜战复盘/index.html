从晚上八点到凌晨 00:20：fluxclash 的 0-1 夜战复盘

发表于2026-03-22|更新于2026-03-23|技术日记项目复盘
|浏览量:

摘要：这篇是一次实打实的“夜战复盘”。我从 2026 年 3 月 22 日 20:00 开工，做一个新项目 fluxclash（基于 Mihomo、使用 Go 开发），一路 Vibe Coding 到 2026 年 3 月 23 日 00:20。最难的不是某一段代码，而是环境、打包、发布、桌面集成和体验优化同时压上来。好消息是：项目已经跨过最难的 0-1，进入可迭代优化阶段。

时间： 2026.03.22 20:00 - 2026.03.23 00:20
关键词： fluxclash / Mihomo / Go / Wails / RPM / CI 发布链路
状态： 累到爆，但主链路打通了

🌙 开局：这次不聊理想，先把项目跑起来

今晚开新坑，目标很明确：做一个基于 Mihomo 的 Go 项目 fluxclash，先把 0-1 跑通。
我给自己的节奏是：先功能闭环，再谈审美和高级体验。

结果从一开始就进入“多线程打怪”模式。
每推进一步，背后都牵着另一条链路：能跑不代表能构建，能构建不代表能发布，能发布不代表装完能在桌面用。

1) 环境与工具链：第一关就不是代码

问题

  • 系统里一开始缺 make
  • go 全局命令不可用

处理

我没继续死磕系统全局环境，而是直接改为项目内 .toolchain 路径执行。
命令统一加 PATH，让工具链就地可控，规避不同机器的差异。

收获

“环境可复现”比“我本机能跑”更重要。
第一关过不去，后面全是幻觉进度。

2) Wails 运行标签：桌面壳不是普通 Go 入口

问题

执行 go run ./cmd/fluxclash-desktop 直接报错：

根因

Wails 的桌面壳运行方式和普通 Go 程序不一样，不能裸跑入口包就完事。

处理

调整运行脚本和命令路径，走正确的桌面启动方式。
把“能启动”这件事先稳定下来，再继续叠功能。

3) 打包与发布链路：有 tag，不等于有产物

问题

Release 页面只有 tag 的源码包，没有实际可用构建产物。

根因（典型）

  • workflow 触发条件不对
  • 权限不完整
  • artifact 上传或发布步骤缺失

处理

持续补齐 CI 发布逻辑，按“构建 -> 上传 artifacts -> 生成 release”做闭环。
目标不是“看起来发布了”，而是“用户真能下载并使用”。

4) RPM 安装后菜单搜不到：桌面集成细节决定生死

问题

rpm 安装成功，但应用菜单里搜不到入口。

根因

.desktop、图标、Exec、分类、安装路径这些桌面集成要素不完整或不规范。

处理

把桌面集成资源补齐，并纳入打包流程，避免“本地能用、安装后失踪”。

5) UI 集成度：功能有了，但体验还是“网页平铺感”

现状

初版能用，但离目标体验差距明显，像一个能跑的网页，而不是成熟桌面应用。

推进路径

我按优先级分三步补闭环：

  1. 先补“导入 YAML 预校验 + 抽屉确认”
  2. 再补“核心启停”
  3. 最后补“连接信息 + 当前配置摘要 + 一键切换配置”

结果

功能闭环完整度明显提升。
视觉和交互还有提升空间，但已经过了“只能演示不能用”的阶段。

6) 权限与沙箱：代码之外的真实阻力

问题

当前会话 writable root 指向 clash_new，实际项目在 /home/rik/fluxclash，导致写入受限，index.lock 报只读。

处理

按需提权做文件编辑和 git 操作。
这类问题和业务逻辑无关，但会直接阻断交付链路，必须正面处理。

7) 命令执行稳定性：先稳态，再并行

问题

并行命令偶发 CreateProcess 异常（尤其带 -lc 或路径组合时）。

处理

关键步骤改为单条稳态命令，先保证结果可交付。
并行是加速器，不是必须项；主链路稳定优先级更高。

8) 今晚最大的流程经验：每一步都做最小闭环

这次能一直推进没有失控，核心不是我写得有多快，而是流程策略对了：

  • 先打通 可运行 + 可构建 + 可发布
  • 再打磨 UI 和交互
  • 每一步都最小验证闭环：接口 -> 前端 -> build/test -> commit/push

这套节奏让“复杂项目的焦虑”变成“可执行的小任务”。

✅ 总评：最难 0-1 已过，项目进入可迭代区间

今晚的最大难点，不是单点编码，而是多维问题并行：

  • 环境
  • 打包
  • 发布
  • 桌面集成
  • 体验优化

好在最关键的拐点已经跨过去。
fluxclash 现在已经不是“想法”，而是一个可持续迭代的工程实体。

到 2026 年 3 月 23 日 00:20，我真的扛不住了，但这次累是值的。
因为我不是在原地焦虑，而是在把复杂系统一点点推到可交付状态。

🔮 下一步

  • 把发布流水线继续标准化，减少手动干预点
  • 深挖性能和资源占用，优化常驻体验
  • 做一轮高质量 UI/交互打磨，拉齐“能用”到“好用”的差距
